---
name: wjt-plan
description: 立项协议——把当前研讨蒸馏成正式任务(研讨Agent本人立项, 上下文零损耗)


你和用户的研讨已收敛, 现在把它变成正式任务。你此刻拥有最完整的讨论
上下文——由你立项, 蒸馏损耗最小。这些上下文会随会话结束蒸发,
现在是浇筑成文件的最后机会。

1. 与用户确认三件事: 任务短名(kebab-case)、执行者(默认你自己)、
   feat还是fix
2. 立项(自动二选一):
   - git status --porcelain 为空(纯嘴聊, 无产物) -> aiwt new <执行者> <任务名>
   - 主工作区有本次研讨产生的未提交文件(风暴框架写的spec/草稿) ->
     aiwt adopt <执行者> <任务名>  (建worktree的同时把产物一并搬走,
     主工作区恢复干净; 这是 adopt 在人类当次指示下的合法使用)
   (你有权自主执行 new/save/when/path/list; adopt 仅限本协议此步,
   done/drop/freeze 仍只能由人类发起)
3. 把研讨成果蒸馏进新worktree的任务纪要, commit:
   - 目标: 一句话
   - 计划: 30-60分钟粒度子任务(fix类只写诊断项, 规矩见 wjt-bugfix)
   - 验收标准+验收命令: 这是研讨中"我们说好的效果"的机器化表达,
     蒸馏的重头戏。讨论里每一句"要能够X"都应对应一条可检验标准
   - 决定记录: 研讨中敲定的技术决定 + 被否掉的方案(含理由), 一行一条
     ——被否的方案就是墓地的种子, 不写下来执行者会再提一遍
   - 研讨产生的长文档: 留在风暴框架的原生目录(openspec/等)或
     docs/<任务名>/ 均可, 只要在任务分支上受版本管理; 纪要放一行指针
4. 汇报一行: "已立项: <worktree路径> ｜ 计划N项 ｜ 验收: <命令>。
   由我继续请说'开工', 或 cd 该目录换其他Agent"

禁止: 立项后未经用户指示就开工; 在主工作区改任何业务代码——
若研讨期间已在main产生代码改动, 提醒用户 aiwt adopt 收编。
