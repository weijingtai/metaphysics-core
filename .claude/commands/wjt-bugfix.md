description: Bug修复协议——先红后绿、范围锁死、历史优先。修复类任务用此协议替代wjt-resume的开工协议
---

你在修一个bug。先执行 /wjt-resume 的恢复步骤(读纪要/git状态/快照),
然后按以下协议干活, 与新功能开发的差异点全在这里:

## 诊断阶段(计划区此时只有诊断子任务, 属正常)
1. 历史优先: aiwt when "<报错关键词/现象词>" —— 病历可能显示这个bug
   被修过又复发, 或墓地里有失败修法。命中则先读对应纪要区块。
2. 先红后绿(不许跳): 把纪要「目标」里的复现步骤写成自动化测试,
   跑一遍【确认它是红的】, commit "test: 复现<现象>(红)"。
   复现不出来 -> 停下, 把尝试过的复现路径写入决定记录并报告, 不许盲修。
3. 定位根因: git log -S / git blame; 需要 git bisect 时允许使用,
   但仅限当前worktree内, 结束必须 git bisect reset(这是"禁止切分支"
   铁律的唯一豁免)。
4. 根因写入「决定记录」: "根因: X, 引入于Y, 机制是Z"。
   然后把修复子任务【补充】进计划区并 commit——诊断结束才有资格写修法。

## 修复阶段
5. 范围锁死: diff只允许触碰根因链路上的代码。禁止顺手重构、禁止
   "既然打开了这个文件就优化一下"。发现更大的结构性问题 -> 写入
   决定记录 + 停在当前范围, 报告调度者另立任务。
6. 修复后: 复现测试必须变绿, 且全量验收命令无新增失败。
7. 无效的修法立即入「踩坑墓地」(哪怕只试了十分钟)。
8. 干别的任务时撞见的无关bug: 决定记录记一行+上报, 不许顺手修。

## 收尾
9. /wjt-handoff 落盘。确认行格式:
   "已落盘: <hash> / 根因: <一句话> / 复现测试: 红->绿 / 新入墓 <N>条"
复现测试永久保留在测试集里——它是这个bug的疫苗。
